# Dogfooding 项目初始化设计

## 背景

当前仓库是一个空 Git 仓库，分支为 `feat-init-repo`，还没有任何提交。这个仓库将作为团队内部 dogfooding 空间，用于孵化早期项目，并在真实内部工作流中验证 `univer-cli` 与 `univer-cli skill` 的使用方式。

第一个预期项目是 `daily report`。它的目标是帮助团队成员快速、结构化地记录每日工作事项，并在晨会中展示和同步。本设计不实现 `daily report`，也不提前定义它的目录、字段或流程；本次只完成启动该仓库前所需的最小文档设计。

## 目标

- 用中文建立仓库定位，服务团队内部协作。
- 保持初始化足够轻量，只做文档骨架。
- 为未来早期项目提供清晰入口。
- 沉淀一个最小项目孵化流程，让后续项目能一致地开始，但不引入重流程。

## 非目标

- 不添加 `package.json`、TypeScript 配置、测试工具、格式化工具或脚本。
- 不创建 `projects/daily-report` 目录。
- 不定义 `daily report` 的字段、表格结构、晨会展示流程或自动化。
- 不要求未来所有早期项目立即遵循统一目录模板。

## 拟新增文件

### `README.md`

`README.md` 是仓库入口，面向第一次打开仓库的人。它将说明：

- dogfooding 仓库是什么，以及为什么存在。
- 当前重点是即将开始的 `daily report` 项目。
- 仓库原则：真实使用、轻量记录、持续演进。
- 当前最小文档结构。
- 下一步是单独设计 `daily report`。

### `docs/project-incubation.md`

该文档面向准备新增早期项目的人。它将说明：

- 哪类项目适合进入这个仓库。
- 新项目开始前最少需要回答的问题：
  - 谁会使用它。
  - 它验证哪条工作流。
  - 它依赖或验证哪些 `univer-cli` 能力。
  - 它预期产出什么记录、文件或工作成果。
  - 团队如何判断继续推进、调整方向或沉淀经验。
- 推荐先写清楚场景和使用方式，再按需要补工具链或自动化。
- `daily report` 是第一个样例，但不是强制模板。

## 数据流

本次初始化没有应用数据流，只有文档使用路径：

1. 团队成员打开 `README.md`，理解仓库目的和当前重点。
2. 团队成员在准备新增早期项目时阅读 `docs/project-incubation.md`。
3. 后续项目文档可以引用孵化流程文档，而不需要改变这次初始化的边界。

## 错误处理

因为本次只做文档，错误处理的重点是避免误导和过早承诺：

- `README.md` 中的链接必须指向实际存在的文件。
- 文档不能暗示 `daily report` 已经实现。
- 项目孵化流程必须表达为轻量指导，而不是严格框架。
- 文档不能留下未完成占位或含糊的待补内容。

## 验证

实现该初始化时需要验证：

- 只新增预期文档文件。
- `README.md` 正确链接到 `docs/project-incubation.md`。
- `README.md` 和 `docs/project-incubation.md` 使用中文。
- 没有引入包管理、代码、工具链或 `daily report` 项目目录。
- 没有未完成占位、范围矛盾或和本设计不一致的表述。

## 完成标准

初始化完成时应满足：

- `README.md` 说明仓库目的，并指向项目孵化文档。
- `docs/project-incubation.md` 说明未来早期项目的轻量进入流程。
- 本次变更保持文档-only。
- 下一步仍然是单独设计 `daily report`。
